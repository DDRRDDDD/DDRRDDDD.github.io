#### 🚨 직면했던 과제
> **"반복적인 문서화 작업으로 인한 몰입 방해와 컨벤션 불일치"**
> * **컨벤션 준수의 피로감**: 커밋 컨밴션을 매번 수동으로 지키는 것은 개발 흐름을 끊는 요소였습니다.
> * **히스토리 품질 저하**: 바쁜 개발 일정 속에서 단순히 "코드 수정"과 같이 모호한 메시지가 남발되거나, 변경의 '이유'가 누락되어 추후 코드 트래킹에 어려움이 있었습니다.

#### 🛠 문제 해결 및 구현 과정
> **💡 GitHub Copilot을 활용한 커밋 메시지 자동 생성 파이프라인**
> * **커스텀 인스트럭션 적용**: `instruction.md`에 정의된 **규칙**을 코파일럿의 시스템 프롬프트로 주입하여, 변경된 코드를 분석해 규칙에 부합하는 메시지를 제안하도록 설정했습니다.
> * **문맥 기반의 'Why' 도출**: 단순히 변경된 코드를 나열하는 것이 아니라, 코파일럿이 수정 전후의 로직을 분석하여 **"왜 이 변경이 필요했는지"를 본문에 포함**하도록 유도해 문서의 깊이를 더했습니다.

#### 📈 성과/개선
> * **개발 생산성 극대화** : 커밋 메시지 고민과 작성에 소요되던 시간을 **0**에 가깝게 단축하여, 로직 구현 자체에만 온전히 몰입할 수 있는 환경을 만들었습니다.
> * **유지보수 효율 향상** : 모든 커밋 로그가 표준화된 양식과 명확한 의도를 갖추게 되어, 협업 시 불필요한 커뮤니케이션 비용을 줄이고 프로젝트의 장기적인 관리 용이성을 확보했습니다.

```markdown
# 코파일럿 코드 커밋 메시지 작성 지침

- 모든 커밋 메시지는 **한국어**로 작성합니다.
- 제목(Subject)과 본문(Body)은 **빈 줄 하나로 구분**합니다.
- 제목은 **50자 이내**, 동사 원형으로 시작하고 요약 중심으로 작성합니다.
- 본문은 **72자 단위 줄바꿈**을 사용합니다.
- 본문에서는 "무엇을(What)" 보다는 "왜(Why)"를 설명합니다.
- 여러 항목은 불릿 포인트(- 또는 *)로 나열합니다.
- 사실만을 기술하고, 주관적인 의견은 피합니다.
- 의미 없는 내용으로 커밋하지 않습니다.
- 코드 가독성 개선 내용은 커밋 메시지에 포함하지 않습니다.

## 커밋 메시지 유형

- feat : 새로운 기능을 추가할 경우
- fix : 버그를 수정한 경우
- style : 코드 포맷팅, 세미콜론 누락, 코드 변경이 없는 경우
- refactor : 코드 리팩토링을 한 경우
- comment : 필요한 주석 추가 및 변경
- docs : 문서(README, WIKI 등)를 수정한 경우
- test : 테스트 추가, 테스트 리팩토링(프로덕션 코드 변경 X)
- chore : 빌드 테스트 업데이트, 패키지 매니저를 설정하는 경우(프로덕션 코드 변경 X)
- rename : 파일 혹은 폴더명을 수정하거나 옮기는 작업만인 경우
- remove : 파일을 삭제하는 작업만 수행한 경우
```
