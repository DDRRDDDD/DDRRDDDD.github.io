#### [개요]
사용자의 프로필 수정, 모임 활동 이력, 앱 설정, 계정 관리 등 다양한 성격의 정보가 집약된 **마이페이지**를 담당했습니다. 단순한 목록 나열이 아닌, 정보의 중요도와 위계(Hierarchy)에 따른 **트리 구조의 UI 아키텍처**를 설계하여 사용자 접근성을 높이는 데 주력했습니다.

#### [직면했던 과제: 복잡한 정보 구조와 네비게이션]
마이페이지는 앱 내에서 가장 정보의 깊이(Depth)가 깊은 곳이었습니다. '마이페이지 > 설정 > 알림 설정' 처럼 화면이 깊어질수록 사용자가 현재 위치를 잃거나, 뒤로 가기 시 원하지 않는 화면으로 이동하는 등의 네비게이션 이슈가 발생하기 쉬웠습니다. 또한, 정적인 정보와 동적인 정보(활동 배지, 레벨 등)가 섞여 있어 효율적인 위젯 배치가 필요했습니다.

#### [상세 구현 과정]
* **정보 위계에 따른 UI 트리 구조화**
    * 사용자가 정보를 직관적으로 탐색할 수 있도록 페이지 구조를 3단계로 기획 및 개발했습니다.
    * **Top Level (Header):** 사용자 프로필 이미지, 닉네임, 핵심 활동 지표(온도, 레벨) 등 가장 자주 확인하는 동적 데이터를 배치했습니다.
    * **Mid Level (Body):** '나의 모임', '관심 카테고리' 등 사용자의 활동 이력을 카드 형태의 리스트(`ListView`)로 구성하여 가독성을 높였습니다.
    * **Low Level (Settings):** 앱 버전, 로그아웃, 회원탈퇴 등 접근 빈도가 낮은 기능은 하위 뎁스로 분리하여 화면 복잡도를 낮췄습니다.

* **네비게이션 라우팅 및 상태 관리**
    * 깊은 뎁스의 페이지 이동을 효율적으로 관리하기 위해 Flutter의 `Navigator` 스택을 체계적으로 관리했습니다.
    * 설정 페이지 진입 후 변경 사항이 저장되면, `pop` 시 파라미터를 전달하여 상위 페이지(마이페이지)가 최신 상태로 갱신되도록 처리했습니다. 이를 통해 불필요한 전체 화면 리빌드를 방지하고 부드러운 화면 전환 경험을 제공했습니다.

* **유지보수를 고려한 컴포넌트 분리**
    * 마이페이지 내의 각 섹션(프로필 카드, 메뉴 리스트 아이템, 설정 토글 버튼 등)을 별도의 커스텀 위젯으로 분리했습니다.
    * 이는 첫 프로젝트였지만 **'재사용 가능한 코드 작성'**을 의식적으로 실천한 것으로, 추후 메뉴가 추가되거나 디자인이 변경될 때 해당 컴포넌트만 수정하면 되는 유지보수의 용이성을 확보했습니다.

#### [성과 및 회고]
초기 기획 단계에서는 모든 정보를 한 페이지에 넣으려 했으나, 이를 **논리적인 트리 구조**로 재설계함으로써 사용성을 크게 개선했습니다. 이 과정을 통해 **"개발자는 단순히 기능을 구현하는 것을 넘어, 사용자가 정보를 어떻게 소비할지 고민해야 한다"**는 UI/UX 관점의 인사이트를 얻을 수 있었습니다.
